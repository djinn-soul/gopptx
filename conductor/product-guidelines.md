# Product Guidelines - gopptx

## Prose Style
- **Technical & Precise:** Documentation should prioritize accuracy, performance metrics, and architectural clarity.
- **Action-Oriented:** Use active voice and imperative mood for tutorials and installation steps.
- **Standardized Terminology:** Consistently use "Go Engine" to refer to the core logic and "Python Bindings" for the high-level API.

## Code Quality Standards
- **Performance First:** All code contributions should consider CPU and memory overhead.
- **Safety:** Go code must adhere to strict typing and error handling; Python bindings must use type hints.
- **Documented API:** Every public function, class, and method must have a docstring or comment explaining its purpose and performance implications.

## Communication Channels
- **GitHub Issues:** The primary hub for bug reports and feature requests.
- **Technical ADRs:** Architecture Decision Records should be used for significant changes to the C bridge or core engine.

## Visual Identity (Placeholders)
- **Logo:** A stylized, minimalist representation of a fast-moving presentation slide or a "G" integrated with a PowerPoint icon.
- **Color Palette:** Professional blues and grays, representing stability and speed.
